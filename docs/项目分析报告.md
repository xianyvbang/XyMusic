# XyMusic 项目分析报告

> 生成日期:2026-06-15
> 范围:全量代码通读(约 900 个 Kotlin 文件,9 个 Gradle 模块)
> 性质:功能说明 + 技术架构 + 问题清单。本文档仅作分析记录,未修改任何代码。

---

## 第一部分:应用概述

XyMusic 是一个 **Kotlin Multiplatform (KMP) + Compose Multiplatform** 音乐客户端,连接用户自建的音乐服务器,从单一共享代码库同时构建 **Android / JVM 桌面 (Windows·macOS·Linux) / iOS** 三端。

核心价值是:**用一套统一的 UI 接入五种不同协议的音乐服务器**——Jellyfin、Emby、Subsonic、Navidrome、Plex,外加自定义媒体 API。

### 模块划分

| 模块 | 职责 | Kotlin 文件数 |
|------|------|------|
| `composeApp` | 应用核心:UI 屏幕、ViewModel、DI 装配、导航、播放器控制器、数据源层 | ~465 |
| `api` | 纯 HTTP/服务器客户端层,基于 Ktor,各服务器一个客户端 | ~212 |
| `ui` | 可复用 Compose 组件、主题、弹窗、占位符(`cn.xybbz.ui.xy` 设计系统) | ~79 |
| `localdata` | 主数据库的 Room 实体与 DAO(`LocalDatabaseClient`)及迁移 | ~77 |
| `xy-database` | 共享 Room 基类(`DatabaseClient`)、平台 `DatasourceFactory`、类型转换器 | ~11 |
| `download` | 自包含下载引擎(调度器、下载器、任务调度、独立 Room 库、通知) | ~48 |
| `xy-platform` | 薄平台抽象(`ContextWrapper`),让公共代码触及 Android `Context` | ~6 |
| `androidApp`/`desktopApp`/`iosApp` | 各平台入口,初始化 Koin 并启动共享 `App` | 1~4 |

所有包都在 `cn.xybbz` 下。

## 第二部分:功能说明

### 1. 多服务器连接管理
- 支持五种服务器协议 + 自定义媒体 API,每种由独立的 `*ApiClient` 实现协议翻译。
- 连接配置(地址、凭据)持久化在 Room 的 `xy_connection_config` 表,凭据加密存储。
- 支持多连接管理、连接切换、媒体库选择(`SelectLibrary`)。
- 启动时自动登录上次连接;401 时自动重新登录。

### 2. 音乐浏览与播放
- **首页聚合**:最常播放、每日推荐、最新专辑、最近播放、歌单等区块。
- **分类浏览**:全部音乐、专辑、艺术家、流派、收藏列表,均支持分页加载、排序、筛选。
- **详情页**:专辑/歌单详情、艺术家详情、流派详情。
- **搜索**:带请求版本号防竞态的搜索实现。
- **播放器**:队列管理、顺序/单曲循环/随机三种播放模式、上一首/下一首、进度拖动、淡入淡出(仅 Android)、定时关闭。

### 3. 歌词
- 加载优先级:文件内嵌歌词 → 音乐服务/自定义 API(顺序可配)。
- 实时高亮当前行,支持歌词偏移(`lrcOffsetMs`)校准。

### 4. 收藏
- **单曲收藏(Music Favorite Toggle)**:从歌曲行/菜单/播放器对单首歌收藏。
- 专辑收藏、艺术家收藏、批量收藏为**独立概念**(领域术语区分严格)。

### 5. 歌单与歌单文件
- **服务端歌单(Playlist)**:服务器侧的歌曲集合。
- **歌单文件(Playlist File)**:本地 `.txt`/`.m3u8` 文件。
- **歌单文件导入**(读取并解析本地文件)与**服务端歌单导入**(用户确认后应用到服务器)是分离的两步。
- 支持歌单文件导出。

### 6. 下载引擎
- 自包含引擎:入队 → 调度(并发上限默认 3) → Ktor 流式下载(支持 Range 断点续传) → 落库。
- Android 使用 WorkManager + 前台服务通知(含暂停/恢复/取消动作按钮);JVM 用协程 Job;iOS 为空实现。
- 下载库与主库刻意解耦,播放时按 `uid` 运行时反查关联。

### 7. 设置
- 连接管理、内存管理、界面设置、语言配置、缓存上限、代理配置、流媒体质量、自定义 API、背景图片、关于、每日推荐、下载、本地音乐等。

### 8. 平台特性
- **播放引擎**:Android 用 Media3 ExoPlayer(独立 `MediaSessionService` 进程),JVM 用 VLCJ,iOS 占位。
- **图片加载**:Sketch(非 Coil)。
- **文件选择**:FileKit 跨平台。
- **JVM 本地反向代理**:`JvmReverseProxyServer`,设计良好(仅绑 127.0.0.1、随机令牌、常量时间比较)。

## 第三部分:技术架构

### 1. 多服务器数据源抽象(核心设计)

支撑"一套 UI 接入五种服务器"的分层:

```
UI (Compose Screen / ViewModel)
   │  只依赖 DataSourceManager,不碰具体服务器
   ▼
DataSourceManager  ← 单一门面(Koin @Single)
   │  dataSourceServerFlow: StateFlow<IDataSourceParentServer?>
   │  connectionIdFlow: StateFlow<Long>
   ▼
IDataSourceParentServer (抽象基类)  ← 域层:登录/分页/收藏/缓存
   │  5 个实现 + RemoteMediator(Paging3) + Room
   ▼
*ApiClient (api 模块)  ← 纯协议层,各包一个 Ktor HttpClient
   │  DefaultApiClient / DefaultParentApiClient 为公共基类
   ▼
TokenServer(单例) + XyAuthenticatedRequestPlugin  ← 认证注入
```

- **读取(分页)**:UI 订阅 Paging `Flow` → `RemoteMediator.load()` 发 HTTP → `withTransaction` 写 Room → Room `PagingSource` 反向驱动 UI。**Room 是唯一真相源**,网络只负责回填。
- **认证注入**:登录后子类生成 token,经 `TokenServer` 发布;`XyAuthenticatedRequestPlugin` 每次请求通过闭包 `tokenProvider()` 实时读取——这是认证热更新生效的机制。
- **切换服务器**:`changeDataSource()` → `release()`(旧服务器关闭 + flow 置 null) → `switchDataSource()`(Koin `@Factory` 取新实例) → `loginConection()`(后台登录)。

### 2. 依赖注入(Koin + 注解 KSP)

- `initKoin()` 的 `modules()` 为空,几乎全靠注解。
- `StartKoinApp` 标注 `@KoinApplication` + `@ComponentScan("cn.xybbz")`,KSP 编译期扫描全包的 `@Single/@Factory/@Singleton/@KoinViewModel` 生成绑定。
- 手写注解模块:`DatabaseModule`、`ApiModule`、`StartupModule`,平台绑定用 `expect class XxxModule` + `actual`。

### 3. 应用启动(分层设计,亮点)

刻意拆成"轻量门闩 + 首帧后置重依赖"两段:
- `StartupInitializer.start()`:IO 上串行 `initSet()`,再并行 `initData()` ‖ `initConfig()`。
- 重依赖(自动登录、播放器恢复、401 重登录)推迟到首页首帧后,经 `withFrameNanos` → `DataSourceBootstrapper.startAfterHomeVisible()`,用 `AtomicBoolean.compareAndSet` 保证只执行一次。

### 4. 本地持久化(两个 Room 数据库)

- **主库 `LocalDatabaseClient`**(version=8):38 张表、18 个 DAO,关联表用复合主键 + 索引,**无外键**(靠手写 `NOT IN` 清理孤儿)。
- **下载库 `DownloadDatabaseClient`**(version=1):仅 `xy_download` 一张表,与主库刻意解耦。
- 两库都继承 `xy-database` 的 `DatabaseClient`,经 `DatasourceFactory`(expect/actual)构建:JVM 用 `BundledSQLiteDriver`,Android `createFromAsset`,iOS 落 `NSDocumentDirectory`。
- DAO 直接返回 Paging3 `PagingSource`,由数据源层包成 `RemoteMediator`。

### 5. 平台抽象(expect/actual)

- 根基是 `xy-platform` 的 `ContextWrapper`:Android 包 `Context`,JVM 暴露 `applicationDirectory`,iOS 空壳。
- 约 30+ 个 expect/actual 抽象点:媒体播放、网络监控、剪贴板/分享、语言管理、数据库构造、下载引擎、Ktor 引擎、代理。
- 注意:**播放器层未用 expect/actual**,而是抽象基类 + 平台子类 + Koin 绑定的 OOP 多态方式。

### 6. 导航(Navigation3)

- 自实现多返回栈模型:`NavigationState` 持有 `backStacks: Map<RouterConstants, NavBackStack>`,每个顶层路由一条独立栈。
- `Navigator` 封装 `navigate/navigateToRoot/goBack`,维护导航变化监听器。
- `RouterCompose` → `NavDisplay` 渲染,每个 entry 有独立 ViewModelStore(Koin `koinViewModel()` 按页面作用域取 VM 的基础)。
- 路由集中定义在 `RouterConstants`(sealed class + `@Serializable`),约 27 个目的地。

### 7. ViewModel 与状态管理

- 两种状态载体并存:Compose 快照状态(`mutableStateOf` 等)与 `StateFlow/Flow`。
- 分页 VM 继承 `PageListViewModel<T>`,核心:`combine(mergeFlow, sortBy).flatMapLatest{ getFlowPageData }.cachedIn`——服务器切换或排序变化都重建 PagingData。
- `PagingUiState` 把 `LoadState` 收敛为 `isInitialLoading/isRefreshing/isEmpty`,供列表组件复用(设计良好的一层)。

### 8. 关键支撑库

kotlinx.serialization(JSON)、cryptography-kotlin(凭据加密)、TinyPinyin(中文转拼音排序)、Media3 / VLCJ(播放)、Sketch(图片)、FileKit(文件)。

## 第四部分:问题清单

> 按严重程度分级。🔴 严重(数据安全/功能失效/崩溃) · 🟠 中等 · 🟡 轻微/健壮性。
> 每条标注了文件位置,便于定位核实。

### 🔴 严重问题

#### A. 数据安全

**A1. 凭据加密形同明文 —— 密钥与密文同行存储**
`common/utils/PasswordUtils.kt:24-36` 每次随机生成 AES-256 密钥,然后把**密钥本身**(Base64)连同密文、IV 一起写进同一行 `xy_connection_config`(`ConnectionConfig.kt:76/80/84`,三者均 TEXT NOT NULL)。任何拿到该表的人(设备数据库、未加密备份、桌面端明文 SQLite、root/越狱)零额外信息即可还原全部服务器密码。这是"编码混淆"而非有意义的加密。
**建议**:密钥移出数据库,改用平台密钥库(Android Keystore / 桌面 OS keyring)或基于用户口令 KDF 派生;密文改 AES-GCM;删除 `key` 字段。

**A2. Ktor 日志级别 HEADERS 泄漏认证头**
`api/client/KtorClient.kt:28` `level = LogLevel.HEADERS` 全局生效,会把 `Authorization`、`X-Emby-Authorization`、`X-Plex-Token`、Navidrome 头写入日志。release 构建应关闭或脱敏。

**A3. 登录响应整体打印(含 token)**
`JellyfinApiClient.kt:260`、`EmbyApiClient.kt:226`、`NavidromeApiClient.kt:216`、`PlexApiClient.kt:379`、`IDataSourceParentServer.kt:231` 均 `logger.info { "返回响应值: $responseData" }`,responseData 含 accessToken/token。应删除或脱敏。

> 补充:Plex token 明文拼入 URL;Android `usesCleartextTraffic="true"` 且无 networkSecurityConfig;AES 用 CBC 而非 AEAD——均为相关的中危项。

#### B. 数据库迁移(可致静默丢数据 / 崩溃)

**B1. 主库破坏性迁移 + 迁移链空洞 → 静默丢数据(含凭据)**
`localdata/LocalDataBaseCreate.kt:14` 启用 `fallbackToDestructiveMigration(true)`,但只注册了 `Migration_4_5..7_8`,**缺 1→2、2→3、3→4**。版本 1/2/3 的存量用户升级到 8 会直接清库,丢失连接配置(含加密凭据)、收藏、播放历史。
**建议**:改为仅 `fallbackToDestructiveMigrationOnDowngrade`,并补齐 1→4 迁移。

**B2. 下载库迁移配置两头不靠 → 未来升版必崩**
`download/DownloadDataBaseCreate.kt` 的 `getRoomDatabase` 未传任何 migration,也没有破坏性兜底。当前 version=1 安全,一旦升到 2,Android 端 `createFromAsset` + 无迁移 → `IllegalStateException` 崩溃。

#### C. 分页加载(功能性失效,影响全部服务器)

**C1. APPEND 末页判断运算符优先级错误**
`DefaultRemoteMediator.kt:58` 与 `ArtistRemoteMediator.kt:70`:
```kotlin
if (remoteKey == null || (remoteKey.nextKey + 1 * state.config.pageSize) >= remoteKey.total)
```
`*` 优先级高于 `+`,实际算的是 `nextKey + pageSize`(页码与条目数混加)。本意应为 `(nextKey + 1) * pageSize >= total`。后果:翻到第 2 页以后过早判定到末尾,**列表无法继续加载**。

**C2. ArtistRemoteMediator 末页硬编码 `true`**
`ArtistRemoteMediator.kt:106` `endOfPaginationReached = true` 写死,**艺术家列表永远只有第一页**。根因是该类未继承 `DefaultRemoteMediator` 而复制了 load 逻辑。改为继承基类可同时消除 C1/C2 与重复 catch。

**C3. REFRESH 末页判断用起始偏移而非末尾偏移**
`DefaultRemoteMediator.kt:91-92` 用 `loadKey * pageSize`(当前页起点)与 total 比较,比正确的 `(loadKey+1)*pageSize` 少一页,每次多发一次空请求才停止。

#### D. 资源泄漏与并发

**D1. 多个 ApiClient 的 release() 不关闭 HttpClient**
`JellyfinApiClient.kt:437-439` 只清字段,不调 `httpClient.close()` 也不 `super.release()`;Emby/Subsonic/Plex 同样。配合 `DefaultApiClient.createHttpClient()`(作者自留 `//todo 注意关闭`)重建时覆盖旧 client 不关闭——**每次切换服务器/重登录都泄漏一个连接池 + 协程**。`CustomMediaApiClient` 是唯一正确关闭的。

**D2. TokenServer 无锁单例,多线程读写认证状态**
`TokenServer` 的 `token/queryMap/headerMap/tokenHeaderName` 均为普通 `var`,无 `@Volatile`。登录线程写、IO 请求线程读,JVM 上可能读到陈旧认证状态。项目他处(ProxyManager)已用 `@Volatile`。

**D3. login() 的 catch 吞掉 CancellationException**
5 个 ApiClient 的 `login()` 统一把任意 `Exception` 转成 `ConnectionException`,**协程取消异常被误报为连接失败**,破坏结构化并发取消语义。应在 catch 开头 `if (e is CancellationException) throw e`。

**D4. 下载进度回调 onProgress 并发竞态**
`DownloadDispatcherImpl.kt:251-266` 进度回调在 `scope.launch(IO)` 里遍历/修改普通 `mutableMap`,**未持 globalMutex**,与 `promoteAndExecute`/`onStateChanged` 并发增删 → `ConcurrentModificationException`;`lastDbUpdateTime` 多协程无锁,节流失效。

#### E. 播放器功能缺陷与平台不一致

**E1. endTime(片尾跳过)被持久化但从不触发跳过**
`MusicCommonController.kt:106-115` 定义并持久化 `endTime`,但全代码库无任何"接近片尾跳到下一首"的消费点。有声小说的"跳过片尾"特性实际不生效(`headTime` 则有被用于 seek)。

**E2. JVM error() 回调后不前进到下一首 → 播放卡死**
`JvmMusicController.kt:161-175` 出错只 `updateState(None)` 不切下一首;Android `onPlayerError` 会 `onSeekToNext()`。JVM 端某首歌 URL 失效时播放链停摆,需手动干预。**平台行为不一致**。

**E3. 文件名清洗不过滤 `..`,存在路径穿越**
`download/utils/FileNameResolver.kt:18` 正则替换了分隔符但不处理 `..`,文件名直接来自服务端歌曲名。建议拒绝 `.`/`..` 开头,并在拼接后校验 `startsWith(下载根目录)`。

### 🟠 中等问题

#### 数据源 / API

- **取消收藏/删除失败仍返回成功**:`unmarkFavoriteItem` 任意异常时 `return true`;Subsonic/Navidrome 的 `removeById/removeByIds` 直接 `return true` 空操作。UI 与服务端不一致。
- **Subsonic 全量列表 totalRecordCount 硬编码 1000000**:配合末页判断会持续请求直到空页,产生大量无效翻页。
- **Jellyfin/Emby 收藏忽略 dataType 参数**:接收却不用,跨实现语义不一致。
- **共享可变状态无同步**:`IDataSourceParentServer.connectionConfig`(:127)被自动登录(IO)、`bindLocalConectionConfig`(主线程)等并发写,无 `@Volatile`/Mutex;`requestReloginOnAppActive()` 存在 TOCTOU,Android onResume 快速触发可能双重登录。
- **Room 事务内发起网络 IO**:`initOtherData()` 在 `withTransaction` 内调 `initFavoriteData`、`ping()` 等,长时间占用写锁,Android 有 ANR 风险。
- **歌词 `it.start!!` 强解包**:`JellyfinDatasourceServer.kt:357`、`EmbyDatasourceServer.kt:434`,服务端返回 `start=null` 时歌词页崩溃,应 `?: 0L`。
- **DataSourceManager 三重 catch 第三块为死代码**:`DataSourceManager.kt:719-723`,"未知错误"日志分类失效。
- **`updateTokenHeaderValue` 中 `httpClient.config{}` 为死代码**:`DefaultApiClient.kt:186-198` 返回新配置但被丢弃。

#### 下载

- **磁盘空间检查全程缺失**:入队、写文件前均无 `usableSpace` 校验,会写满磁盘后才失败。
- **Android 最终路径与库记录不一致**:MediaStore 写入返回 `displayName`,但落库用解析阶段的私有路径,本地播放会失败。
- **删除歌曲/清库后下载文件残留**:`DatabaseUtils.kt:65` 只删库记录,绕过引擎 `delete(deleteFile=true)`,磁盘文件永久残留。
- **播放前不校验本地文件是否存在**:`MusicPlayAssembler.kt:106-114` 只要 status=COMPLETED 就贴 filePath,手删文件后会播放不存在路径且无在线回退。
- **临时文件写入未 fsync,无校验**:崩溃时尾部可能丢失,迁移到正式目录前无 size/MD5 校验。
- **JVM 两库都建在 java.io.tmpdir**:系统清理临时目录后记录全失。

#### 类型转换 / 数据库

- **StringListTypeConverter 用逗号分隔损坏含逗号的艺术家名**:`"Earth, Wind & Fire"` 会被错拆,round-trip 不可逆。名称类字段应改 JSON。
- **IntListTypeConverter.toInt() 无容错**:空串/脏数据抛 `NumberFormatException`。
- **核心关联表零外键**:引用完整性全靠手写 `NOT IN` 清理,任一删除路径遗漏即累积孤儿行。
- **索引覆盖不足**:关联表排序键 `index` 列普遍无索引,`year`(被 between 过滤)、`cachedAt`、`lastPlayedDate` 等均无索引,大列表走全表。
- **AlbumDao 疑似复制粘贴 bug**:`AlbumDao.kt:184-194` `selectArtistIndex` 的 SQL 与 `selectHomeIndex` 完全相同(查 homealbum),疑应查 artistalbum;`selectGenrePlayAlbum` 同类问题。需复核。

#### 播放器

- **Android progressTicker 为 lateinit,清理/关闭时可能未初始化**:`MusicController.kt:70`,在异步 `onControllerReady()` 才赋值,若提前触发 `clearPlayerList()`/`close()` 会抛 `UninitializedPropertyAccessException`,缺 `isInitialized` 防护。
- **定时关闭"曲目结束"用脆弱启发式**:`MusicBottomMenuViewModel.kt:304-322` 用"进度变小"判断结束,用户手动 seek 回退会误判为结束而提前停止并 `clearPlayerList()`。
- **PlaybackProgressReporter 作用域永不销毁**:`@Singleton` 自建 scope 无 `close()`;`start()` 与 `reportNow()` 对同一 job 无同步,存在竞态。
- **JVM sekTo 当 millSeconds<=0 静默无操作**:`JvmMusicController.kt:292`,拖到最开头(0)不 seek、不发事件、暂停态不恢复。平台不一致。
- **推荐数据三个并发 launch 无互斥**:`PlayerEventCoordinator.kt:202-218`,`recommendationStateFlow` 经典 read-modify-write 竞态。
- **PlayProgressTicker 每 tick 清空整个 Handler 队列**:`removeCallbacksAndMessages(null)` 会误清共享线程上其他组件的回调。

#### UI / 导航

- **NavigationState.toEntries() 无 remember,每次重组重建集合**:`NavigationState.kt:90-108` `flatMap + toMutableStateList`,放大主壳重组(作者已用 SideEffect 日志在追这个问题)。

#### 其他

- **readyForContent 门闩未被路由消费**:首页可能早于缓存就绪渲染(逻辑竞态)。
- **NetWorkMonitor 共享状态无同步**:`listeners` 裸 list、`started` 裸 Boolean,在轮询线程/dispatch_queue 触发,非原子。

## 第五部分:整体评价与修复优先级

### 设计亮点
- `DataSourceManager` 单门面 + `IDataSourceParentServer` 抽象,把"五种协议"收敛到一个清晰的接缝。
- `PageListViewModel` + `PagingUiState` 两层抽象,统一分页与加载态。
- 启动分层(轻量门闩 + 首帧后置重依赖),避免阻塞 `onCreate`。
- `PlayerEventCoordinator` 事件总线集中处理播放副作用。
- JVM 本地反向代理的安全实现规范(本地绑定 + 随机令牌 + 常量时间比较 + 无证书绕过)。

### 修复优先级总览

排序依据三个维度:**影响面**(数据安全 > 数据丢失 > 功能失效 > 体验/健壮性)、**触发概率**(必现 > 高频 > 边界)、**修复成本/风险**(同等收益下优先做低风险改动)。建议按下面 7 个批次推进,每批次内部相对独立、可单独验证后合入。

### 批次 1 — 数据安全(最高优先级)

这批不影响功能表现,但泄漏的是用户的服务器密码,一旦发布就难以补救(已发出的明文凭据无法收回),所以排在最前。

**1.1 凭据密钥移出数据库 + 改 AES-GCM(对应 A1)**
- **根因**:`PasswordUtils.encryptAES` 每次随机生成 AES-256 密钥,却把密钥本身 Base64 后和密文、IV 写进同一行 `xy_connection_config`。攻击者拿到这张表就同时拿到锁和钥匙,加密退化为编码混淆。
- **改法**(分两步,降低迁移风险):
  1. 先把算法从 AES-CBC 换成 **AES-GCM**(带完整性校验,防密文篡改),保持密钥仍在库内,验证加解密 round-trip 正常。
  2. 再把密钥迁出数据库:
     - **Android**:用 Android Keystore 生成/保管一把主密钥(`KeyGenParameterSpec`,`setUserAuthenticationRequired(false)` 以免后台自动登录被挡),数据库里只存被主密钥包裹后的密文。
     - **桌面**:走 OS keyring(Windows DPAPI / macOS Keychain / Linux Secret Service),或在无 keyring 环境下用「用户口令 + PBKDF2/Argon2 派生密钥」方案,绝不再落盘明文密钥。
     - 删除 `ConnectionConfig` 的 `key` 字段(需配套数据库迁移,见批次 2)。
- **迁移注意**:存量用户库里已有旧格式凭据。需要写一次性迁移:用旧逻辑解出明文 → 用新逻辑重新加密 → 回写,再清掉 `key` 列。务必先做好该迁移再删字段,否则老用户登录态全失。
- **风险**:中高。涉及凭据读写主链路 + 数据库结构变更,改坏会导致所有人需要重新登录。建议独立分支、加单元测试覆盖「旧格式可解、新格式可解、迁移幂等」。
- **验证**:新建连接→重启 App→自动登录成功;升级路径(旧库→新库)登录成功;直接查 SQLite 确认库内不再含可还原的明文密钥。

**1.2 release 构建关闭/脱敏 Ktor 日志(对应 A2)**
- **根因**:`KtorClient.kt:28` 全局 `LogLevel.HEADERS`,把 `Authorization`/`X-Emby-Authorization`/`X-Plex-Token` 等写进日志。
- **改法**:按构建类型区分日志级别——release 用 `LogLevel.NONE`(或 `INFO` 但加 `sanitizeHeader { it == HttpHeaders.Authorization }` 之类的脱敏过滤器屏蔽认证头);debug 保留 `HEADERS` 便于排查。可通过 BuildConfig / expect-actual 的「是否 debug」标志注入级别。
- **风险**:低。仅改日志配置。
- **验证**:release 包运行后检查 logcat / 桌面日志文件不含认证头明文。

**1.3 删除打印登录响应的 logger(对应 A3)**
- **根因**:5 处 `logger.info { "返回响应值: $responseData" }`,responseData 含 accessToken/token。
- **改法**:直接删除这些行,或改为只打印不含敏感字段的摘要(如 `userId`、HTTP 状态)。同时把 Plex token 从拼接进 URL 的日志路径里摘掉。
- **风险**:低。
- **验证**:全局搜索 `返回响应值` 确认清零;登录流程日志不含 token。

> 关联中危项(同批次顺手处理):Android `usesCleartextTraffic="true"` 建议改为按域名/网段白名单的 `networkSecurityConfig`(自建服务器走 HTTP 是现实需求,但不应全局放开);`StringUtils` 默认补 `http://` 可考虑改为优先 `https://`。

### 批次 2 — 数据库迁移(防静默丢数据 / 崩溃)

紧跟批次 1,因为 1.1 删 `key` 字段本身就要走迁移,两批可以合并设计迁移脚本。

**2.1 主库补齐迁移链 + 收窄破坏性策略(对应 B1)**
- **根因**:`LocalDataBaseCreate.kt:14` 开了全局 `fallbackToDestructiveMigration(true)`,但只注册 4→5…7→8,缺 1→2、2→3、3→4。版本 1/2/3 的老用户升到 8 时直接清库,连服务器凭据一起丢。
- **改法**:
  1. 把 `fallbackToDestructiveMigration(true)` 改成 `fallbackToDestructiveMigrationOnDowngrade(true)`——只在「降级」时才允许清库,正常升级绝不静默删数据。
  2. 补齐 `Migration_1_2`、`Migration_2_3`、`Migration_3_4`(现有 `Migration_1_2` 是空壳且未挂载,需要按 1→4 之间真实的表结构差异补 `ALTER TABLE`)。要确认 1/2/3 版本到底改了哪些表——如果已无早期版本用户、且 schema 历史不可考,至少保留破坏性兜底但**加日志/提示**让用户知情。
- **风险**:中。迁移脚本写错会在升级时崩溃。务必用 Room 的 `MigrationTestHelper` + 导出的 schema json(项目已 `exportSchema=true`)做迁移测试。
- **验证**:用各历史版本的 schema 建库 → 跑迁移 → 断言升级后数据完整、无清库。

**2.2 下载库加迁移兜底(对应 B2)**
- **根因**:`DownloadDataBaseCreate.kt` 的 `getRoomDatabase` 既没传 migration 也没破坏性兜底。当前 version=1 安全,但一旦升到 2,Android 端 `createFromAsset` + 无迁移会抛 `IllegalStateException` 崩溃。
- **改法**:现在就补上 `fallbackToDestructiveMigration`(下载任务记录可接受清空,代价低)或预留 `addMigrations` 入口。下载库数据是「可重建」的(重新下即可),用破坏性兜底是合理权衡,但要在升版那一刻就生效,不能等崩了再补。
- **风险**:低。
- **验证**:本地把 version 临时 +1 跑一次升级,确认不崩。

### 批次 3 — 分页加载失效(用户直接可感知,改动集中)

这批是「列表加载不全」的根因,用户每天都会碰到,但修复代码量极小、风险低,性价比最高。

**3.1 修正 APPEND 末页判断运算符优先级(对应 C1)**
- **根因**:`(remoteKey.nextKey + 1 * state.config.pageSize)` 因 `*` 优先级高于 `+`,实际算成 `nextKey + pageSize`(页码混加条目数),翻到第 2 页后过早判定到底。
- **改法**:加括号改为 `(remoteKey.nextKey + 1) * state.config.pageSize >= remoteKey.total`。
- **风险**:低。但要同步检查 `nextKey` 语义到底是「页码」还是「偏移量」,确保公式与之匹配(这是该 bug 的本质——量纲混用)。

**3.2 ArtistRemoteMediator 改继承基类(对应 C2,顺带消除 C1/C3 重复)**
- **根因**:`ArtistRemoteMediator` 没继承 `DefaultRemoteMediator`,复制了一份 load 逻辑,且把 APPEND 的 `endOfPaginationReached` 写死 `true`,导致艺术家列表永远只有第一页。
- **改法**:让它继承 `DefaultRemoteMediator`,只重写「取数据」的差异点。这样 3.1 的修复自动生效,也不再有第二份重复的末页/catch 逻辑。
- **风险**:中(改了继承结构)。需回归艺术家列表的首屏 + 翻页 + 下拉刷新。

**3.3 修正 REFRESH 末页判断偏移(对应 C3)**
- **根因**:`loadKey >= 1` 时用 `loadKey * pageSize`(当前页起点)比 total,比正确的 `(loadKey+1) * pageSize` 少一页,每次都多发一次空请求。
- **改法**:统一成「末尾偏移」口径 `(loadKey + 1) * pageSize >= total`。
- **风险**:低。
- **批次验证**:对每种服务器(Jellyfin/Emby/Subsonic/Navidrome/Plex)的音乐列表、专辑列表、艺术家列表分别滚动到底,确认能加载全部页且不重复请求空页。

### 批次 4 — 资源泄漏与并发正确性

这批不影响首次使用,但随使用时长/切换次数累积,表现为「越用越卡」「偶发认证错乱」,排查成本高,值得在功能稳定后专门处理。

**4.1 release() 统一关闭 HttpClient(对应 D1)**
- **根因**:Jellyfin/Emby/Subsonic/Plex 的 `release()` 只清字段不调 `httpClient.close()`,加上 `createHttpClient()` 重建时直接覆盖旧 client(作者已留 `//todo 注意关闭`),每次切服务器/重登录泄漏一个连接池 + 协程。
- **改法**:在各 `release()` 里调用 `httpClient.close()` 并 `super.release()`;`createHttpClient()` 重建前先关旧实例。可参考 `CustomMediaApiClient`(唯一正确的)抽一个基类统一处理,避免 5 份各写各的。
- **风险**:中。要确保 close 后不再有 in-flight 请求复用旧 client(close 时机应在确认无活跃请求后)。
- **验证**:反复切换服务器 N 次,用内存/线程分析确认连接池与协程数不持续增长。

**4.2 TokenServer 加 @Volatile(对应 D2)**
- **根因**:`token/queryMap/headerMap/tokenHeaderName` 是裸 `var`,登录线程写、IO 请求线程读,JVM 上可能读到陈旧值。
- **改法**:给这些字段加 `@Volatile`;Map 类字段改为引用整体替换(发布一个新的不可变 Map 而非原地改),保证读到的是一致快照。项目他处 ProxyManager 已是这个模式,照搬即可。
- **风险**:低。
- **验证**:切换服务器后立刻发起请求,确认带的是新认证信息。

**4.3 login() 放行 CancellationException(对应 D3)**
- **根因**:`catch (e: Exception)` 把协程取消异常也转成 `ConnectionException`,切换服务器取消登录时被误报为「连接失败」,破坏结构化并发。
- **改法**:在每个 `login()` 的 catch 开头加 `if (e is CancellationException) throw e`(5 处统一)。
- **风险**:低。
- **验证**:快速连续切换服务器,确认不弹出虚假的连接失败提示。

**4.4 下载进度回调统一加锁(对应 D4)**
- **根因**:`DownloadDispatcherImpl.kt:251-266` 的 `onProgress` 在 `scope.launch(IO)` 里遍历/修改普通 `mutableMap`,未持 `globalMutex`,与 `promoteAndExecute`/`onStateChanged` 并发增删会抛 `ConcurrentModificationException`;`lastDbUpdateTime` 无锁导致节流失效。
- **改法**:进度读写统一进 `globalMutex`,或把 `runningTasks` 换成 `ConcurrentHashMap` + 落库时取不可变快照;`lastDbUpdateTime` 改 `@Volatile` 或 atomic。
- **风险**:中。锁粒度要小心,别把进度回调阻塞在长事务上。
- **验证**:并发下载多个大文件 + 中途暂停/取消,确认不崩、进度落库正常。

### 批次 5 — 播放器功能缺陷与平台一致性

功能性缺陷,但只在特定场景(有声小说、出错歌曲、桌面端 seek 到 0)触发,且 Android/JVM 行为不一致,需要分别处理。

**5.1 实现 endTime 片尾跳过(对应 E1)**
- **根因**:`endTime` 被定义、写入、持久化,但全代码库没有「播放到 endTime 就切下一首」的消费点,「跳过片尾」实际不生效。
- **改法**:在进度更新处(Android 的 `PlayProgressTicker` tick / JVM 的 `positionChanged` 回调)加判断:当 `endTime > 0 && 当前位置 >= endTime` 时触发 `onSeekToNext()`。注意与 `headTime`(已生效的片头 seek)对称实现,并处理单曲循环模式下的语义。
- **风险**:低中。要避免与自然结束/手动 seek 冲突触发两次切歌。
- **验证**:给一首歌设 endTime,播放到该点自动切下一首。

**5.2 JVM 播放出错后前进到下一首(对应 E2)**
- **根因**:`JvmMusicController.error()` 只 `updateState(None)` 不切歌,URL 失效时播放链停摆;Android `onPlayerError` 会 `onSeekToNext()`。
- **改法**:在 JVM `error()` 里也调 `onSeekToNext()`(或带重试上限的跳过逻辑,避免整列表都失效时无限切歌)。
- **风险**:低。
- **验证**:桌面端播放一个失效 URL,确认自动跳到下一首而非卡死。

**5.3 JVM sekTo 处理 millSeconds<=0(对应 E2 相关)**
- **根因**:`JvmMusicController.kt:292` 的 `if (millSeconds > 0)` 导致拖到最开头(0)时不 seek、不发事件、暂停态不恢复。
- **改法**:把条件放宽到 `>= 0`,与 Android 行为对齐。
- **风险**:低。
- **验证**:桌面端把进度条拖到 0,确认从头播放且事件正常。

**5.4 下载文件名路径穿越加固(对应 E3)**
- **根因**:`FileNameResolver.kt:18` 正则替换了分隔符但不处理 `..`,文件名直接来自服务端歌曲名。
- **改法**:拒绝 `.`/`..` 开头的名字(或替换为安全占位),并在拼出最终路径后用规范化路径校验 `startsWith(下载根目录的规范化绝对路径)`,不满足则拒绝。
- **风险**:低。属安全加固,建议尽早做。
- **验证**:构造歌名为 `../../evil` 的下载任务,确认文件落在下载目录内或被拒绝。

### 批次 6 — 崩溃防护(低成本、高保险)

这批改动都极小(几行),但能挡掉真实的崩溃路径,适合随手搭车做。

- **progressTicker 加 `isInitialized` 防护**:`MusicController.clearPlayerList()`/`close()` 访问 `lateinit progressTicker` 前先判 `::progressTicker.isInitialized`,避免 MediaController 异步连接完成前触发清理时抛 `UninitializedPropertyAccessException`。
- **初始播放索引用 `getOrNull`**:`JvmMusicController.initMusicList` 的 `musicDataList[targetIndex]` 改为 `getOrNull(targetIndex)`,历史恢复索引越界时不崩。
- **歌词 `it.start!!` 改 `?: 0L`**:`JellyfinDatasourceServer.kt:357`、`EmbyDatasourceServer.kt:434`,服务端返回 `start=null` 时歌词页不崩。
- **IntListTypeConverter.toInt() 加容错**:空串/脏数据时返回空列表而非抛 `NumberFormatException`。

每项验证就是构造对应的边界输入,确认不再崩溃。

### 批次 7 — 数据一致性与体验优化

不影响「能不能用」,但影响「用得对不对/顺不顺」,放在功能稳定后逐步清理。

- **收藏/删除返回真实结果**:`unmarkFavoriteItem` 等不要在异常时 `return true`;Subsonic/Navidrome 的 `removeById/removeByIds` 要么真正实现删除、要么明确返回 false 并在 UI 提示「该服务器不支持删除」。
- **清库联动删下载文件**:`DatabaseUtils.kt:65` 删库记录时改走下载引擎的 `delete(deleteFile=true)`,避免磁盘残留。
- **播放前校验本地文件存在**:`MusicPlayAssembler` 贴 `filePath` 前确认文件还在,不在则回退到在线流地址。
- **Room 事务内剥离网络 IO**:`initOtherData()` 把 `ping()`、`initFavoriteData` 等网络调用移到事务外,先取数据再开短事务写库,降低 Android ANR / 写锁竞争。
- **NavigationState.toEntries() 加 remember + 首页连接菜单 remember**:缓解主壳重组(作者已用 SideEffect 日志在追这个性能问题)。
